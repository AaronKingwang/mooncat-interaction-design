# B端产品交互设计通用 Skill

> 一套可对照、可执行、可评分的 B端交互设计工作法。

> **作者声明**：本 skill 由 **MoonCat** 原创整理。其中的方法论框架、B端场景落地、评审打分体系、清单与模板均为作者原创编写；所参考的交互设计思想方向归各原书作者所有（见下方参考书目）。

> **来源声明**：本 skill 的内容是**参考**上述四本交互设计经典中广为人知的核心思想，并结合 B端实践整理、推演而成，**并非对原书逐页精读后的原文提炼**。文中若提及某本书/某位作者，均表示「参考其思想方向」，不代表是书中原话或可逐条对应到具体章节。需要严格溯源时，请回到原书核对。

## 触发场景

当用户出现以下任意意图时启用本 skill：

- **设计新页面/交互**：列表页、表单页、详情页、仪表盘、工作流、筛选、批量操作、权限、引导、空状态、反馈
- **索要设计输入**：交互原则、模式参考、命名与布局建议、组件选型
- **评审已有方案**：对原型/设计稿/HTML/交互说明做「交互质量评审」，输出问题清单与改进建议
- **讨论可用性问题**：「这个交互合不合理」「怎么设计更好用」「为什么用户会卡住」「这里该怎么引导」

### 高频触发词（优先激活）

> 以下原话出现时，优先使用本 skill：
> - "帮我设计一个 B端..."
> - "这个表单/列表/详情页怎么设计"
> - "评审一下这个交互"
> - "为什么用户在这里会迷路/卡住"
> - "B端设计规范/原则"
> - "怎么减少操作步骤"

### 本 skill 的边界（不深入这些领域）

- **纯视觉/UI 风格**（配色、字体、插画、动效）：本 skill 只判断它是否影响交互可用性，不输出视觉规范
- **前端代码实现**：本 skill 只到交互设计/原型建议层面，不生成具体组件代码
- **C端/消费端产品**：若用户明确是 C端场景且与 B端效率无关，仅提供通用可用性建议，不强行套用 B端模式

> B端 ≠ C端。本 skill 的一切结论都以 **B端特征** 为前提：专家/高频用户、效率优先于新奇、数据密集、流程复杂、错误代价高、需长时间停留（主宰姿态 sovereign）。

## 一、四条核心信条（参考四本书的思想方向）

> 下表左列是本 skill 参考的书，中列是据其思想方向整理的一句话要点（**为便于记忆的概括，非原文**），右列是在 B端的落地含义。

| 参考来源               | 核心信条（整理概括）                     | 在 B端的含义                                                         |
| ------------------ | ------------------------------ | --------------------------------------------------------------- |
| 《About Face》Cooper | **为用户的目标而设计，而非为任务或功能而设计**      | 别堆功能，先问「用户真正想达成什么」；消灭一切「操作杂物(excise)」——用户为使用软件本身而付出、却不产生业务价值的动作 |
| 《Web界面设计》Tidwell   | **不要发明轮子，复用成熟的界面模式**           | B端有成熟模式库（表格、筛选、向导、面板）。用对模式=用户零学习成本                              |
| 《简约至上》Colborne     | **简单来自「合理地分配复杂度」，而非删掉复杂度**     | 用「删除/组织/隐藏/转移」四策略管理 B端不可避免的复杂性                                  |
| 《点石成金》Krug         | **别让我思考（Don't make me think）** | 每个页面/控件应「一目了然」；用户不读、只扫；满意即走(satisfice)，不追求最优解                   |

**四句合一的设计信条**：
> 面向目标 → 复用模式 → 分配复杂度 → 让它无需思考。

## 二、B端交互设计四步法（Goal-Directed 精简版）

参考「目标导向设计（Goal-Directed Design）」的思想方向，为 B端裁剪：

1. **定角色与目标**：谁用？他的岗位目标（体验目标/最终目标/人生目标）是什么？高频动作是哪几个？
   - B端务必区分角色（操作员/审核员/管理员/老板看板），不同角色的界面密度与权限完全不同。
2. **理清主线场景**：写出「一天中最常做的 1-3 条主路径」的文字剧本。主线决定默认视图、默认排序、默认按钮。
3. **选模式、排布局**：为每个页面挑成熟界面模式（查 `references/designing-interfaces.md`），按视觉层级摆放（查 `references/dont-make-me-think.md`）。
4. **分配复杂度、去杂物**：用四策略（查 `references/simple-and-usable.md`）把复杂度藏到该在的地方，逐条消灭 excise（多余点击/翻页/等待/记忆负担）。

> 每一步都问 Krug 三连：**这是什么？我在哪？我能点什么？** 如果 3 秒内答不上来，就是设计缺陷。

## 三、简约至上·四策略（B端复杂度的分配引擎）

面对「功能太多、页面太满」时，按顺序尝试：

1. **删除（Remove）**——去掉不解决核心目标的功能/字段/选项。B端最常见的浪费：把「后台能查的字段」全塞进列表。
2. **组织（Organize）**——按用户心智分组、排序、分区。用「分块、对齐、就近」组织信息，而非按数据库表结构组织。
3. **隐藏（Hide）**——把低频/专家功能收进「更多/高级/展开」，但保证需要时找得到。B端典型：高级筛选、批量操作二级菜单。
4. **转移（Displace）**——把复杂度转移到更合适的地方：给系统（智能默认值、自动计算）、给另一端（把配置搬到设置页）、给用户（仅对专家开放的自定义）。

> 顺序有意义：**先删，删不掉再组织，组织不完再隐藏，隐藏不了才转移。** 详见 `references/simple-and-usable.md`。

## 四、B端通用交互原则清单（对照使用）

参考上述效率与可用性思想整理，设计/评审时逐条对照：

**效率与去杂物（Excise）**
- [ ] 完成主任务的点击/翻页/等待是否已压到最少？高频操作能否一步直达？
- [ ] 是否消除了「无意义的导航跳转」（弹窗套弹窗、来回翻页找信息）？
- [ ] 常用参数是否有智能默认值？用户是否被迫重复输入系统已知的信息？
- [ ] 支持键盘操作/批量操作/记住上次选择吗？（B端高频用户强需求）

**清晰与无需思考（Don't make me think）**
- [ ] 每个按钮/链接的文案是否「自解释」？（用「导出Excel」而非「操作」）
- [ ] 页面有明确的视觉层级吗？最重要的东西是否最显眼？
- [ ] 用户永远知道「我在哪、我能去哪、我刚做了什么」吗？（面包屑/高亮/反馈）
- [ ] 是否遵循平台约定俗成（叉号关闭、蓝色主按钮、表格在右、导航在左）？别自创。

**反馈与容错**
- [ ] 每个操作是否有即时反馈（loading/成功/失败）？失败是否说清原因+下一步？
- [ ] 破坏性操作（删除/批量/不可逆）是否有确认 or 可撤销(Undo)？优先「可撤销」而非「问一遍」。
- [ ] 错误信息是否人话、可行动，而非「Error 500 / 系统异常」？
- [ ] 是否用「防错」优于「报错」——从源头让用户填不错（禁用非法项、格式约束、范围限制）？

**模型一致（心智模型 vs 实现模型）**
- [ ] 界面呈现的是「用户心智模型」还是「系统实现模型」？别把数据库结构直接搬到界面。
- [ ] 术语用业务语言，不用技术黑话（用「已发货」而非「status=3」）。

**状态完整**
- [ ] 空状态、加载中、无权限、超长内容、异常数据 五种边界态是否都设计了？（详见 `references/bend-scenarios.md`）

## 六、两种使用姿势

### 姿势 A：设计模式（边设计边对照）

1. 先跑「四步法」（第二节）确认角色、主线、模式、复杂度分配。
2. 针对当前页面类型，读 `references/bend-scenarios.md` 对应场景的要点。
3. 输出设计建议时，**给方案 + 给依据**（说明参考的是哪条设计原则），并列出「至少 2 个候选 + 推荐项」。
4. 交付前用第四节「通用原则清单」自查一遍。

### 姿势 B：评审模式（对已有原型/设计打分）

1. **先理解业务上下文再评审**：评审前先确认「谁用、什么岗位、什么频率、要达成什么、当前处于设计哪个阶段、有什么业务/技术约束」。若用户未说明，主动追问；不臆断。
2. 读 `references/review-checklist.md` 的评分维度。
3. 逐维度检查目标物（原型 HTML / 设计稿描述 / 交互说明）。
4. 输出「问题清单」：每条含 **【严重度】+ 位置 + 问题 + 参考原则 + 改进建议**。
5. 严重度分级：`阻断`（用户无法完成任务）>`严重`（明显低效/易错）>`一般`（体验瑕疵）>`建议`（可优化）。
6. **给具体方案，而非只列问题**：针对 Top 问题输出 **2–3 个可落地的原型/交互优化方案**，每个方案含「适用条件、优劣、推荐度」，供用户选择后再细化。
7. **面向产品经理输出**：用流程描述、线框说明、决策表、交互规则来表达方案，**不输出前端代码**（HTML/Vue/React/JS 等）。若用户明确要开发交付物，再单独说明。
8. 给总分 + Top3 必改项，避免一次抛出几十条让人无从下手（这本身就违反「别让我思考」）。

## 七、故障与边界处理

| 触发条件 | 一线处理 | 仍失败兜底 |
|---------|---------|-----------|
| 用户没说清角色/场景，无法判断 B端 vs C端 | 先问「谁用、什么岗位、什么频率、要达成什么」；不臆断 | 若用户不答，默认按「高频专家用户+主宰姿态」给建议，并显式标注该假设 |
| 需求要求违反可用性原则（如「删除不要确认，也不要撤销」） | 指出风险（引用具体原则），给「既满足诉求又控风险」的折中方案 | 用户坚持则执行，但在交付物中标注「[交互风险：{简述}，已按用户要求保留]」 |
| 场景没有现成可参考的对应模式 | 拆解为已知子模式的组合；回到第一性原则（目标/去杂物/别让我思考）推导 | 无成熟模式时，明确告知「此为新模式，建议做可用性测试验证」，给最小可测方案 |
| references 分册读取失败 | 用本 SKILL.md 第三、四节的精简清单作为 fallback 继续 | 覆盖不了的深度问题，标注「[需查证：{书名+主题}]」，不编造书中细节 |
| 评审时目标物信息不足（只有截图/一句话） | 说明「基于现有信息，仅能评审 X 维度」，列出需补充的材料 | 不足以评审的维度明确留白，不硬凑评分 |

## 八、核心原则

1. **目标先于功能**：永远先问「用户要达成什么」，再谈怎么做界面。
2. **上下文先于判断**：评审或设计方案前，先理解业务角色、主流程、使用频率和约束条件；信息不足时主动追问，不凭假设下结论。
3. **给方案，不给清单**：无论是设计还是评审，最终交付物都应是用户可选的**具体原型/交互方案**（含优劣对比与推荐），而不是抽象问题列表。
4. **面向产品经理，不写代码**：用流程描述、信息架构、页面结构、交互规则、边界条件来表达方案；不输出 HTML/Vue/React/JS 等实现代码，除非用户明确索要开发交付物。
5. **复用胜于发明**：优先用成熟模式，自创交互必须有充分理由并验证。
6. **分配而非删除复杂度**：B端复杂性无法消灭，只能合理安放。
7. **别让用户思考**：一目了然、自解释、符合约定，是 B端效率的地基。
8. **给依据**：每条设计/评审建议都要能追溯到一条设计原则，不凭感觉（原则参考自上述经典，但以原则本身立论，不假称出自某书原文）。
9. **少而准**：评审时聚焦 Top 必改项，不用长清单淹没用户。
