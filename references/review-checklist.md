# 交互质量评审 · 打分清单与模板

> 用于「姿势 B：评审模式」。对已有 B端原型/设计稿/HTML 做结构化交互评审，输出可执行、有据、分轻重的问题清单。
> **来源声明**：下表「参考方向」列中的作者名，仅表示该维度**参考了谁的思想方向**，不代表评分标准出自其书原文。

---

## 1. 评审七维度（按权重合计 100 分）

| #   | 维度          | 权重  | 核心问题                                   | 参考方向           |
| --- | ----------- | --- | -------------------------------------- | -------------- |
| 1   | **目标契合**    | 20  | 页面是否服务于用户真实目标？默认视图/排序/按钮是否对齐主任务？       | Cooper 目标导向    |
| 2   | **效率/去杂物**  | 20  | 完成主任务的点击/翻页/等待是否最少？高频操作是否直达？有无重复输入？    | Cooper Excise  |
| 3   | **清晰/无需思考** | 15  | 3秒能看懂吗？可点元素明显吗？文案自解释吗？符合约定吗？           | Krug           |
| 4   | **模式/一致性**  | 10  | 是否复用成熟模式？同类元素是否一致？有无为创新反直觉？            | Tidwell        |
| 5   | **复杂度分配**   | 10  | 是否合理删除/组织/隐藏/转移？主流用户是否被专家功能干扰？         | Colborne       |
| 6   | **反馈/容错**   | 15  | 每操作有反馈吗？防错优于报错吗？危险操作可撤销/有确认吗？错误信息可行动吗？ | Cooper         |
| 7   | **边界状态**    | 10  | 空/加载/错误/无权限/极端数据 五态是否齐全？               | Tidwell/Cooper |

> 评审信息不足时，只评能评的维度，其余维度标注「材料不足，未评」，不硬凑分。
>
> **关于分数的说明**：七维度打分是为了结构化定位问题、对齐优先级，不是精确度量。LLM 在计算总分、理解权重或跨维度比较时可能出现偏差，最终结论应结合业务上下文、实际用户反馈与可用性测试综合判断。不要纠结 5 分以内的波动，重点看严重度分布和 Top 必改项。

## 2. 逐维度检查项（勾选式）

**① 目标契合**
- [ ] 页面能一句话说清「用户来做什么」
- [ ] 默认排序/筛选/视图服务最高频任务
- [ ] 区分了不同角色的目标与视图

**② 效率/去杂物**
- [ ] 主任务点击/翻页压到最少，高频操作直达
- [ ] 无重复输入系统已知信息；有智能默认值
- [ ] 保持上下文（操作后不丢筛选/不跳首页）
- [ ] 支持批量/键盘/记住上次选择（高频场景）

**③ 清晰/无需思考**
- [ ] 3秒测试通过（能说出这是什么页、重点、能干嘛）
- [ ] 按钮/链接文案是动词、说清后果（非「操作/确定」）
- [ ] 视觉层级分明，重要的先被看到
- [ ] 遵循约定（图标/位置/颜色语义），无反直觉创新
- [ ] 面包屑/返回/当前位置清晰（我在哪、怎么回去）

**④ 模式/一致性**
- [ ] 用了对的成熟模式（表格/看板/向导/主从…）
- [ ] 同类元素样式、位置、交互一致
- [ ] 未把数据库结构直接搬成界面结构

**⑤ 复杂度分配**
- [ ] 无冗余字段/列/选项（该删的删了）
- [ ] 信息按用户心智分组（组织）
- [ ] 低频/专家功能已折叠（隐藏），入口可发现
- [ ] 复杂度合理转移（系统默认/设置页/专家自定义）

**⑥ 反馈/容错**
- [ ] 每个操作有 loading + 成功/失败反馈
- [ ] 可逆操作用「执行+撤销」，未滥用确认框
- [ ] 不可逆/高危操作有确认且写清影响范围
- [ ] 防错优于报错（约束输入、禁用非法项）
- [ ] 错误信息人话、可行动，非「系统异常/500」

**⑦ 边界状态**
- [ ] 空状态：有原因+下一步
- [ ] 加载中：loading/骨架/进度
- [ ] 错误态：清晰可行动
- [ ] 无权限：隐藏或置灰+原因
- [ ] 极端数据：超长/超多/异常值有处理

## 3. 问题清单输出模板

每条问题按此结构写，便于团队直接排期修复：

```
【严重度】维度 · 位置 —— 问题描述
  参考原则：{设计原则}
  建议：{具体改法}
```

**严重度分级**
- `阻断`：用户无法完成核心任务（必修，最高优先）
- `严重`：明显低效或高易错，日复一日放大（应修）
- `一般`：体验瑕疵，不致命（可修）
- `建议`：优化项，锦上添花（选修）

**示例**
```
【严重】效率 · 工单列表 —— 审批需进详情页点3次才能完成，日均数百次
  参考原则：消灭操作杂物（excise）—— 高频操作应直达
  建议：列表行内增加「通过/驳回」快捷按钮，支持批量审批；保留详情页做复杂case

【严重】反馈 · 批量删除 —— 删除无确认也无撤销，误删不可恢复
  参考原则：撤销优于确认 + 危险操作需明确影响范围
  建议：删除后 toast 提供「撤销」；若不可逆，确认框写明「将永久删除N条，不可恢复」

【一般】清晰 · 状态列 —— 显示 status=3，用户不懂
  参考原则：勿暴露实现模型 + 别让我思考
  建议：改为业务语言「已完成」，配状态色 Tag
```

## 4. 评审结论模板

```
## 交互评审结论：{页面/原型名}

总分：XX / 100
- 目标契合 X/20 · 效率去杂物 X/20 · 清晰无需思考 X/15
- 模式一致 X/10 · 复杂度分配 X/10 · 反馈容错 X/15 · 边界状态 X/10

### Top 3 必改（阻断/严重）
1. …
2. …
3. …

### 其余问题（一般/建议）
- …（可折叠，避免一次淹没）

### 做得好的地方
- …（正向反馈，指出可保持的设计）
```

> 评审的可用性原则：**聚焦 Top 必改，先解决阻断与严重项**。一次抛几十条会让接收方「无从下手」——这本身就违反「别让我思考」。
